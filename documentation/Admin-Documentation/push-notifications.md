# Push Notifications

## Overview

The push notification system enables real-time alerts when activities are logged and timer-based reminders when feed/diaper thresholds are exceeded. Subscriptions are per-baby, per-device and work on Chrome, Firefox, Edge, and Safari 16.4+.

Push notifications require HTTPS in production and must be enabled through the Family Manager settings page.

**Important**: The `ENABLE_NOTIFICATIONS` environment variable only controls whether the notification infrastructure (cron daemon, log directories) is built into the application. It does not turn on notifications. The database always defaults to having notifications **disabled**. You must enable them in the Family Manager settings page after **SSL certificates** have been configured.

## Setup

### Step 1: Build with Notification Support

#### Docker

The included `docker-compose.yml` sets `ENABLE_NOTIFICATIONS=true` by default, which builds the cron daemon and log directories into the container. The container startup script automatically:

- Generates `NOTIFICATION_CRON_SECRET` if missing
- Sets up the cron job for timer checks
- Starts the cron daemon

Set `APP_URL` or `ROOT_DOMAIN` in your environment so the cron job can reach the notification API:

```bash
APP_URL=https://baby.example.com
```

If neither is set, the cron job defaults to `http://localhost:3000`.

To verify the cron job is running:

```bash
./scripts/docker-setup.sh notification-status
```

#### Local

1. Add to your `.env` file:
   ```bash
   ENABLE_NOTIFICATIONS=true
   NOTIFICATION_CRON_SECRET=<auto-generated or run ./scripts/env-update.sh>
   APP_URL=http://localhost:3000
   ```

2. Set up the cron job:
   ```bash
   npm run notification:cron:setup
   ```

3. Restart the application.

### Step 2: Configure SSL

Push notifications require HTTPS in production. Set up SSL certificates for your domain before enabling notifications.

### Step 3: Enable in Family Manager

1. Log in to the Family Manager at `/family-manager`
2. Navigate to Settings button (a App Configuration form will popout from the right side of the screen)
3. Scroll to "Push Notifications"
4. VAPID keys should already be present from database seeding (or click "Generate New Keys" to rotate)
5. Enable notifications and save

### Per-User Setup

Once notifications are enabled in the database users will be prompted by modal that they can ignore or enable notifications. These notifications are setup per device per family. In the event a user ignores or skips the enable notifications popup they can enable notifications by following these instructions:

1. Navigate to Settings
2. Scroll to "Push Notifications"
3. Click "Enable Notifications"
4. Grant browser permission when prompted
5. Configure preferences per baby

## Configuration

### Database-Managed Settings (Admin UI)

Most settings are stored in the `NotificationConfig` database table and managed via App Configuration:

- **Enable Notifications** -- runtime on/off toggle
- **VAPID Public Key** -- public key for push subscriptions
- **VAPID Private Key** -- stored encrypted with AES-256-GCM
- **VAPID Subject** -- mailto: URI identifying the application (auto-generated)
- **Log Retention Days** -- days to retain notification logs (default: 30, range: 1-365)

### Environment Variables

See [Environment Variables](environment-variables.md) for the full reference. Key notification variables:

- `ENABLE_NOTIFICATIONS` -- build-time infrastructure toggle
- `NOTIFICATION_CRON_SECRET` -- secures the cron endpoint
- `APP_URL` / `ROOT_DOMAIN` -- base URL for cron API calls

### How the Two Flags Work

The notification system uses a dual flag approach:

- **Build-time** (`ENABLE_NOTIFICATIONS` env var): Controls whether the cron daemon and log directories are included in the build. This only makes the infrastructure available -- it does not enable notifications.
- **Runtime** (`NotificationConfig.enabled` in the database): Controls whether notification APIs are active. This is toggled in the Family Manager settings page and defaults to **off**.
- When disabled at runtime, all notification APIs return 503
- The cron shell script checks `/api/notifications/cron-enabled` before proceeding

## VAPID Key Management

VAPID keys are auto-generated in the database during initial seeding. They can also be rotated via the admin UI.

**Important**: Rotating VAPID keys invalidates all existing push subscriptions. Users must re-subscribe after key rotation.

Keys are stored encrypted in the database (AES-256-GCM using `ENC_HASH`).

## Timer Notifications

### How It Works

1. Cron job runs every minute (`* * * * *`)
2. Queries all enabled timer-type preferences
3. For each baby, checks last activity time against the configured threshold
4. Sends notifications to eligible subscriptions
5. Updates `lastTimerNotifiedAt` for notified preferences
6. Runs cleanup operations (failed subscriptions, old logs)

### Repeat Intervals

Timer notifications support repeat intervals:

- **Once per expiration** (default) -- notify once when timer expires, wait for new activity
- **Every 15 minutes**
- **Every 30 minutes**
- **Every hour**
- **Every 2 hours**

## Security

### Cron Secret

The `NOTIFICATION_CRON_SECRET` protects the timer check endpoint:

- Must be non-empty for timer notifications to work
- Recommended: 32+ characters (64 hex characters = 32 bytes)
- Auto-generated by `env-update.sh` using `openssl rand -hex 32`
- Uses timing-safe comparison to prevent timing attacks

### Input Validation

- Endpoints must be valid HTTPS URLs (localhost exempt in development)
- Activity types validated against known enum values
- Timer intervals must be one of: 15, 30, 60, 120 minutes (or null)
- Subscription keys are Base64-encoded, max 512 characters

## Cleanup and Maintenance

### Automatic Cleanup

During each cron execution:
- Subscriptions with 5 or more failures are removed
- Notification logs older than the retention period are deleted

### Manual Cleanup

Trigger cleanup manually by calling the cron endpoint with the correct bearer token, or run:

```bash
npm run notification:cron:run
```

## API Reference

### Public Endpoints

| Method | Path | Description |
|--------|------|-------------|
| GET | `/api/notifications/vapid-key` | Returns public VAPID key (no auth required) |
| GET | `/api/notifications/cron-enabled` | Returns whether cron is enabled (no auth required) |

### Authenticated Endpoints

| Method | Path | Description |
|--------|------|-------------|
| POST | `/api/notifications/subscribe` | Register a push subscription |
| DELETE | `/api/notifications/subscribe?endpoint=...` | Remove subscription by endpoint |
| GET | `/api/notifications/subscriptions` | List all devices for current user |
| DELETE | `/api/notifications/subscriptions/[id]` | Remove a specific device |
| GET | `/api/notifications/preferences` | Get notification preferences |
| PUT | `/api/notifications/preferences` | Update notification preferences |
| POST | `/api/notifications/cron` | Trigger timer check (requires `NOTIFICATION_CRON_SECRET`) |
| POST | `/api/notifications/generate-vapid` | Generate new VAPID keypair (sysadmin only) |
| GET | `/api/notifications/status` | Notification system status (sysadmin only) |

## Troubleshooting

- **Notifications not sending**: Check browser console, verify service worker is registered, confirm VAPID keys are configured in admin UI
- **Cron job not running**: Verify `ENABLE_NOTIFICATIONS=true`, check cron daemon is started, review cron logs
- **Timer notifications not working**: Confirm cron job is set up, check that timer preferences are enabled for the baby
- **After VAPID key rotation**: All users must re-subscribe to notifications
