## Overview

This document defines an external API for Sprout Track that enables integration with third-party services and automation platforms. The API is consumer-agnostic â€” any service that can make HTTP requests can use it: Home Assistant, Grafana, Node-RED, n8n, IFTTT, custom scripts, cron jobs, etc.

The API supports two primary directions of data flow:

- **GET** â€” External services poll Sprout Track for current state and recent activity (dashboards, sensors, monitoring)
- **POST** â€” External services push new events into Sprout Track (physical buttons, automations, NFC tags, voice assistants)

Authentication is handled via API keys scoped to a family. Users can create as many API keys as they want â€” one per service, one per automation, one per physical button, whatever makes sense for their setup.

## Authentication Model

### API Key Design

Each API key is scoped to a **family** and optionally restricted to a specific **baby**. Keys are generated by the account holder from the Sprout Track settings UI.

```
Authorization: Bearer st_live_<random_32_char_hex>
```

Prefix convention:
- `st_live_` â€” production key
- `st_test_` â€” test key (optional, future consideration)

### Schema Addition

```prisma
model ApiKey {
  id          String    @id @default(cuid())
  name        String    // User-provided label: "Nursery Button", "HA Dashboard", etc.
  key         String    @unique // st_live_<32 hex chars>
  keyHash     String    // SHA-256 hash of the key (for lookup after creation)
  familyId    String
  babyId      String?   // Optional: restrict to a specific baby
  scopes      String    // JSON array: ["read", "write"] or ["read"] or ["write"]
  lastUsedAt  DateTime?
  expiresAt   DateTime? // Optional expiration
  revoked     Boolean   @default(false)
  createdAt   DateTime  @default(now())
  updatedAt   DateTime  @updatedAt

  family Family @relation(fields: [familyId], references: [id])
  baby   Baby?  @relation(fields: [babyId], references: [id])

  @@index([keyHash])
  @@index([familyId])
  @@index([revoked])
}
```

**Key storage approach:** The full key is shown to the user exactly once at creation time (like a Stripe API key). Only the hash is stored in the database. Lookups use the hash. The first 8 characters (`st_live_XXXXXXXX...`) are stored in plaintext as a prefix for identification in the UI.

### Key Scopes

| Scope | Allows |
|-------|--------|
| `read` | GET endpoints â€” fetching activity data, summaries, baby info |
| `write` | POST endpoints â€” creating new activity records |

A key can have both scopes or just one. A "dashboard-only" key would be read-only. A "nursery button" key would be write-only scoped to a single baby.

## Base URL & Versioning

```
/api/hooks/v1/...
```

All external integration endpoints live under `/api/hooks/` with explicit versioning. A future `/api/hooks/v2/` can ship breaking changes while v1 stays intact. The `hooks` namespace keeps these cleanly separated from internal app routes.

### Common Response Envelope

```json
{
  "success": true,
  "data": { ... },
  "meta": {
    "timestamp": "2026-03-09T14:30:00Z",
    "familyId": "clx...",
    "babyId": "uuid..."
  }
}
```

Error responses:

```json
{
  "success": false,
  "error": {
    "code": "INVALID_ACTIVITY_TYPE",
    "message": "Unknown activity type: 'nap'. Valid types: sleep, feed, diaper, mood, note, pump, play, bath, measurement, medicine"
  }
}
```

These are designed for any service that polls an API on an interval â€” Home Assistant's RESTful Sensor, Grafana JSON data sources, monitoring dashboards, custom scripts, etc.

### GET `/api/hooks/v1/babies/:babyId/activities`

Returns recent activity logs with optional filtering. For consumers that want richer history or specific activity types.

**Query parameters:**

| `limit` | int | 10 | Max records (1â€“50) |
| `since` | ISO datetime | 24h ago | Only return records after this time |

## POST Endpoints (Write Scope)

These allow external automations, physical buttons, NFC tags, and scripts to log events directly into Sprout Track.

### POST `/api/hooks/v1/babies/:babyId/activities`

Creates a new activity record. The `type` field determines the required/optional fields.

**Common fields (all types):**

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `type` | string | yes | Activity type (see below) |
| `time` | ISO datetime | no | Defaults to now. Useful for backdating. |
| `caretakerName` | string | no | Matched to an existing caretaker by name (case-insensitive). If omitted, logged without caretaker. |

#### Feed

```json
{
  "type": "feed",
  "feedType": "BOTTLE",
  "amount": 4,
  "unitAbbr": "OZ",
  "bottleType": "formula",
  "notes": "Finished the whole bottle"
}
```

| Field | Type | Required | Notes |
|-------|------|----------|-------|
| `feedType` | `BREAST` / `BOTTLE` / `SOLIDS` | yes | |
| `amount` | float | no | Required for BOTTLE |
| `unitAbbr` | string | no | Default from family settings |
| `side` | `LEFT` / `RIGHT` | no | For BREAST only |
| `bottleType` | string | no | `formula`, `breast milk`, `formula\breast`, `milk`, `other` |
| `food` | string | no | For SOLIDS only |
| `notes` | string | no | |

#### Diaper

```json
{
  "type": "diaper",
  "diaperType": "WET"
}
```

| Field | Type | Required | Notes |
|-------|------|----------|-------|
| `diaperType` | `WET` / `DIRTY` / `BOTH` | yes | |
| `condition` | string | no | |
| `color` | string | no | |
| `blowout` | boolean | no | Default: false |

#### Sleep

```json
{
  "type": "sleep",
  "sleepType": "NAP",
  "action": "start"
}
```

| Field | Type | Required | Notes |
|-------|------|----------|-------|
| `sleepType` | `NAP` / `NIGHT_SLEEP` | yes | |
| `action` | `start` / `end` / `log` | yes | `start` begins a new sleep, `end` closes the active one, `log` creates a completed record |
| `duration` | int (minutes) | no | Required with `action: "log"` |
| `location` | string | no | |
| `quality` | `POOR` / `FAIR` / `GOOD` / `EXCELLENT` | no | |

The `start`/`end` pattern is important for sleep because HA users will likely have two button presses â€” one to start, one to end. `log` is for after-the-fact recording with a known duration.

#### Mood

```json
{
  "type": "mood",
  "mood": "HAPPY",
  "intensity": 4
}
```

#### Note

```json
{
  "type": "note",
  "content": "First time rolling over!",
  "category": "milestone"
}
```

#### Pump

```json
{
  "type": "pump",
  "action": "start"
}
```

Or for a completed pump log:

```json
{
  "type": "pump",
  "action": "log",
  "duration": 20,
  "leftAmount": 3,
  "rightAmount": 2.5,
  "unitAbbr": "OZ",
  "pumpAction": "STORED"
}
```

#### Bath

```json
{
  "type": "bath",
  "soapUsed": true,
  "shampooUsed": true,
  "notes": "Loved the rubber duck today"
}
```

#### Measurement

```json
{
  "type": "measurement",
  "measurementType": "WEIGHT",
  "value": 18.5,
  "unit": "LB"
}
```

#### Medicine

```json
{
  "type": "medicine",
  "medicineName": "Tylenol",
  "amount": 1.25,
  "unitAbbr": "ML",
  "notes": "For teething"
}
```

Medicine is a bit special because it references the `Medicine` model. The API should match by name (case-insensitive) against existing medicines for the family. If no match is found, return an error with the available medicine names rather than auto-creating.

### Response (all POST)

```json
{
  "success": true,
  "data": {
    "activityType": "feed",
    "id": "new-uuid",
    "time": "2026-03-09T14:30:00Z",
    "details": { ... }
  }
}
```

## Integration Examples

### Home Assistant â€” Physical Buttons & NFC

One of the most compelling use cases is **physical buttons in the nursery**. Here are some Home Assistant configurations this API enables:

**IKEA/Zigbee button on the changing table** â€” single press logs a wet diaper, double press logs dirty, long press logs both:

```yaml
automation:
  - alias: "Diaper - Wet (single press)"
    triggers:
      - trigger: device
        device_id: zigbee_button_01
        type: remote_button_short_press
    actions:
      - action: rest_command.sprout_track_diaper
        data:
          diaperType: "WET"
```

**NFC tag on the crib** â€” tap phone to start/stop sleep:

```yaml
automation:
  - alias: "Toggle Sleep"
    triggers:
      - trigger: tag
        tag_id: nfc_crib_tag
    actions:
      - action: rest_command.sprout_track_sleep_toggle
```

**Automation: announce last feed time on smart speaker when entering the nursery:**

```yaml
automation:
  - alias: "Nursery Feed Reminder"
    triggers:
      - trigger: state
        entity_id: binary_sensor.nursery_motion
        to: "on"
    conditions:
      - condition: numeric_state
        entity_id: sensor.jackson_last_feed
        above: 120
    actions:
      - action: tts.speak
        data:
          message: "Jackson's last feed was {{ states('sensor.jackson_last_feed') }} minutes ago"
```

### Grafana â€” Monitoring Dashboard

The `/status` and `/activities` endpoints work as JSON data sources in Grafana. Poll the status endpoint to build panels showing feed intervals, diaper counts over time, and sleep duration trends. The `dailyCounts` object maps naturally to Grafana stat panels and time series.

### Node-RED / n8n â€” Workflow Automation

Any workflow engine that can make HTTP requests can use these endpoints. Examples: log a feeding when a specific phrase is spoken to a voice assistant, send a Slack message to a partner when a diaper is logged, or aggregate daily stats into a Google Sheet at midnight.

### cURL / Cron / Scripts

The simplest integration â€” a shell script on a schedule:

```bash
# Poll status and pipe to jq for a quick terminal dashboard
curl -s -H "Authorization: Bearer st_live_abc123..." \
  http://sprout-track.local:3000/api/hooks/v1/babies/uuid-1/status | jq '.data.dailyCounts'

# Log a feeding from the command line
curl -X POST -H "Authorization: Bearer st_live_abc123..." \
  -H "Content-Type: application/json" \
  -d '{"type": "feed", "feedType": "BOTTLE", "amount": 4, "unitAbbr": "OZ"}' \
  http://sprout-track.local:3000/api/hooks/v1/babies/uuid-1/activities
```

| Scope | Limit |
|-------|-------|
| GET (per key) | 60 requests/minute |
| POST (per key) | 30 requests/minute |
| Global (per family) | 120 requests/minute |

These are generous for typical use cases. A dashboard polling every 60 seconds is 1 req/min, and button presses are a handful per hour at most.

Rate limit headers:

```
X-RateLimit-Limit: 60
X-RateLimit-Remaining: 58
X-RateLimit-Reset: 1709985600
```

## What's Deferred (v2 Considerations)

### PUT â€” Editing Existing Records

You mentioned this and I think you're right to defer it. The main use case would be ending an active sleep session or updating a feed amount, but the `action: "end"` pattern on POST handles the sleep case already. A true PUT would need record IDs, partial update semantics, and conflict resolution â€” more complexity than v1 warrants.

If you do add it later, it would look like:

```
PUT /api/hooks/v1/babies/:babyId/activities/:activityId
```

With a body containing only the fields to update. The `activityType` would be inferred from the record.

### DELETE â€” Removing Records

Similar reasoning. Logging errors happen, but they can be fixed in the Sprout Track UI. Exposing delete via API adds risk without much HA-specific value.

### Outgoing Webhooks (Sprout Track â†’ External Services)

The inverse direction: Sprout Track fires a POST to a user-configured URL whenever an event is logged (from any source â€” UI, API, or other caretakers). This is the "push" model that lets external services react in real-time without polling.

This is arguably more powerful than the GET polling approach for real-time automations (e.g., Home Assistant announcing a feed on a smart speaker, or a Slack notification when a caretaker logs a diaper), but it requires a webhook management UI, retry logic, and failure handling on the Sprout Track side. Worth doing in v2.

### Event Stream / WebSocket

For truly real-time dashboards. Overkill for v1 but could be compelling if the HA community engagement is strong.

## Implementation Notes

### Middleware Stack

```
Request â†’ Rate Limiter â†’ API Key Auth â†’ Scope Check â†’ Baby Access Check â†’ Handler
```

The auth middleware extracts the bearer token, hashes it, looks up the `ApiKey` record, verifies it's not revoked/expired, checks the scope matches the HTTP method (GET â†’ read, POST â†’ write), and if the key is baby-scoped, verifies the `:babyId` param matches.

### Settings UI

The API key management UI should live under **Settings â†’ Integrations** (or similar) and support:

- Create key (name, scope, optional baby restriction, optional expiration)
- View existing keys (showing prefix, name, scopes, last used, created date)
- Revoke key
- Copy key (only at creation time)

### Documentation Page

Consider a `/api/docs` or `/integrations/home-assistant` page that:

- Explains the setup with copy-pasteable HA YAML snippets
- Shows the user's actual baby IDs and names (so they don't have to look them up)
- Has a "test" button that fires a sample GET request with their key and shows the response
